### 4. ECS Migration: Always-on from EC2 Docker to ECS Fargate

**What**: Replaced the EC2 Docker-based always-on agent (`docker run` via SSM) with proper ECS Fargate tasks.

**Architecture change**:
- Before: Admin Console → SSM send-command → `docker run` on EC2 (localhost:18800-18899, hardcoded instance ID, no lifecycle management)
- After: Admin Console → `ecs.run_task()` → Fargate task self-registers private VPC IP in SSM → Tenant Router routes to `http://{task_ip}:8080`

**CloudFormation additions** (`clawdbot-bedrock-agentcore-multitenancy.yaml`):
- `AlwaysOnEcsCluster` (Fargate capacity provider)
- `AlwaysOnTaskDefinition` (512 CPU / 1024 MB, awsvpc, ARM64)
- `AlwaysOnEcsTaskRole` (ECR + S3 + SSM + Bedrock + DynamoDB + CW Logs)
- `AlwaysOnTaskSecurityGroup` (inbound 8080 from EC2 SG only)
- `AlwaysOnLogGroup` (/ecs/{stack}-always-on, 30d retention)
- Outputs: cluster name, task def ARN, subnet ID, SG ID

**main.py**: Removed `_next_available_port()`, rewrote `start/stop/status_always_on_agent` for ECS. Added `_get_ecs_config()` helper.

**entrypoint.sh**: After server.py is healthy, reads private IP from `ECS_CONTAINER_METADATA_URI_V4` and registers `http://{ip}:8080` in SSM.

### 5. EFS Persistent Workspace for Always-on Containers

**What**: Added EFS (Elastic File System) as the workspace backend for always-on ECS containers, replacing the S3 sync watchdog (which caused ~1440 unnecessary LIST API calls/day for idle containers).

**Why**: S3 sync every 120s = 1440 API calls/day per container regardless of activity. EFS writes are immediately durable (NFS write-through), zero API overhead when idle.

**CloudFormation additions**:
- `AlwaysOnEFS` (encrypted, elastic throughput, lifecycle IA after 7 days)
- `AlwaysOnEFSMountTarget` (PublicSubnet, same as ECS tasks)
- `AlwaysOnEFSSecurityGroup` (allow NFS 2049 from EC2 + ECS task SGs)
- `AlwaysOnEFSIngressFromECS` (separate ingress rule to avoid circular reference)
- Task definition: EFS volume at `/mnt/efs`, `EFS_ENABLED=true` env var

**entrypoint.sh changes**:
- EFS mode: workspace at `/mnt/efs/{BASE_TENANT_ID}/workspace/`
- Bootstrap: on first start, sync from S3 if EFS directory is empty
- Watchdog: skipped in EFS mode (writes are durable on EFS)
- SIGTERM: S3 snapshot of memory + MEMORY.md + HEARTBEAT.md only (for cross-mode handoff to serverless AgentCore)

**Deployment note**: SSM VPC Endpoint SG (`sg-09cc6fc3f47e02fa8`) needs inbound HTTPS 443 from ECS task SG. Applied manually (not CFn-managed). Documented in README Step 4.5.

### 9. Reminder/Heartbeat Architecture Analysis and CHANNELS.md

**Finding**: OpenClaw has TWO scheduling systems:
1. **CronService** (`~/.openclaw/cron/jobs.json`) — JavaScript `croner` library, in-process, requires persistent Gateway
2. **HEARTBEAT.md** — polling mechanism, LLM checks file periodically, requires persistent Gateway

Both require Gateway to be always running. AgentCore microVMs terminate after each response — no persistent process — so reminders don't work in serverless mode.

**Product decision**: Position scheduled tasks as an always-on feature. Serverless agents acknowledge reminders but note they'll fire on next message. Inject guidance via global SOUL.md.

**CHANNELS.md**: Moved generation to `workspace_assembler.py` (step 7) so always-on containers have it from cold start, before the first user message. Reads from SSM user-mapping reverse lookup. Without this, always-on agents couldn't find the user's IM channel for reminder delivery.

#### Fixed: SOUL.md snowball effect
- `workspace_assembler.py` reads current SOUL.md as "personal layer". On EFS, current SOUL.md is the assembled version from the previous session (global + position + personal + KB). Re-merging produces: 3KB → 11KB → 71KB snowball.
- 71KB SOUL.md: company directory gets buried at the end, beyond effective context.
- Fix: Use `.personal_soul_backup.md` as the canonical personal layer. Backup written on first assembly, never modified. Prevents accumulation.

#### Fixed: Workspace files all MISSING (most critical)
- `server.py _invoke_openclaw_once`: checks `os.geteuid() == 0 and os.path.isdir("/home/ubuntu")`. The Python base image has an ubuntu user with `/home/ubuntu`. Container runs as root → condition TRUE → uses `sudo -u ubuntu HOME=/home/ubuntu`.
- OpenClaw config (`openclaw.json`) was written at startup by `entrypoint.sh` with `HOME=/` to `/.openclaw/openclaw.json`. When running as ubuntu via sudo, openclaw looks for config at `/home/ubuntu/.openclaw/` (missing). Result: no config, no workspace, generic Claude identity.
- Fix: Added ECS detection: `in_ecs = bool(os.environ.get("ECS_CONTAINER_METADATA_URI_V4"))`. When in ECS, skip sudo entirely. `run_env = env` passes all vars including `OPENCLAW_WORKSPACE`, `HOME=/`, etc.

#### Fixed: Gateway reads stale SOUL.md
- Even with correct workspace, OpenClaw Gateway may cache SOUL.md from container startup (before workspace assembly completes). The Gateway runs at T=0; workspace assembly completes at T=30s. New sessions created after T=30s use the Gateway's cached (empty/stale) system prompt.
- Fix: After workspace assembly, mirror assembled files to Gateway's default workspace path: `/root/.openclaw/workspace/`. The Gateway reads from `HOME/.openclaw/workspace/` = `/root/.openclaw/workspace/`. By mirroring there, every new session gets the current assembled identity.

## Key Architectural Insights Discovered

### Always-on ECS Container Environment
```
HOME=/        (root user in container, not /root despite being root)
/.openclaw/   (openclaw config written here by entrypoint.sh)
/root/.openclaw/workspace/  (Gateway reads SOUL.md from here)
/mnt/efs/{emp_id}/workspace/  (EFS workspace with actual SOUL.md)
OPENCLAW_WORKSPACE=/mnt/efs/{emp_id}/workspace/
```

The fundamental mismatch: Gateway reads from `/root/.openclaw/workspace/` but SOUL.md is assembled to `/mnt/efs/{emp_id}/workspace/SOUL.md`. Resolved by mirroring.

### OpenClaw Gateway Session Lifecycle
- Gateway reads SOUL.md at session CREATE time (not at Gateway startup)
- BUT: the workspace path it uses is determined by `OPENCLAW_WORKSPACE` at Gateway startup
- Changes to SOUL.md after a session starts don't affect that session (session already has its system prompt)
- New sessions DO pick up the latest SOUL.md (if mirrored correctly)

### CronService vs HEARTBEAT.md
- Both require the Gateway daemon to be continuously running
- AgentCore microVMs terminate after each response — no persistent daemon
- Always-on ECS containers: Gateway runs 24/7 → both mechanisms work
- Product decision: scheduled tasks are an always-on feature

### S3 vs EFS for Always-on
- S3 sync: ~1440 LIST API calls/day even for idle containers; 60s+ latency before data is durable
- EFS: writes durable immediately; zero API overhead; S3 only used for cross-mode handoff
- Cost: EFS One Zone ~$0.016/GB/month; trivial for workspace data
